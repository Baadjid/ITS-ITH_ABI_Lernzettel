🎯 Was ist Qualitätssicherung?

Die Grundlagen verstehen - Von der Theorie zur Praxis

📖 Definition: Qualitätssicherung (QS)

Qualitätssicherung umfasst alle geplanten und systematischen Tätigkeiten, die sicherstellen, dass ein Produkt, eine Software oder ein System die gestellten Anforderungen erfüllt und fehlerfrei funktioniert.

Einfach erklärt: Stell dir vor, du backst einen Kuchen. Qualitätssicherung wäre, wenn du:

  • ✅ Das Rezept vorher überprüfst
  • ✅ Die Zutaten kontrollierst
  • ✅ Während des Backens die Temperatur überwachst
  • ✅ Den fertigen Kuchen testest, bevor Gäste kommen

🧠 Wichtige Begriffe für die Prüfung

Begriff Definition Beispiel
Verifikation "Bauen wir das Produkt richtig?" Code entspricht den Spezifikationen
Validation "Bauen wir das richtige Produkt?" Software erfüllt Benutzerbedürfnisse
Black-Box Test Testen ohne Code-Kenntnisse Eingabe → erwartete Ausgabe
White-Box Test Testen mit Code-Kenntnissen Alle Pfade im Code durchlaufen
Test Coverage Anteil getesteter Code 80% der Codezeilen wurden getestet
Definition of Done Kriterien für "fertige" Arbeit Code + Tests + Review + Dokumentation

🎯 Warum ist Qualitätssicherung wichtig?

💰 Kostenersparnis

Fehler frühzeitig zu finden ist 100x günstiger als sie später beim Kunden zu beheben.

😊 Kundenzufriedenheit

Zufriedene Kunden kaufen wieder und empfehlen das Produkt weiter.

🛡️ Risikominimierung

Verhindert teure Rückrufaktionen und Imageschäden.

⚖️ Rechtssicherheit

Erfüllung gesetzlicher Vorschriften und Standards.

🌟 Beispiel aus der Praxis

Szenario: Eine Banking-App wird entwickelt.

Ohne QS: App geht online → Benutzer können keine Überweisungen tätigen → Millionen Verlust → Vertrauensverlust

Mit QS: Alle Funktionen werden getestet → Fehler werden vor Release gefunden → App funktioniert perfekt → Zufriedene Kunden

🔍 Verifikation - "Bauen wir das Produkt richtig?"

Verifikation überprüft, ob das System nach den technischen Spezifikationen entwickelt wurde.

💡 Einfaches Beispiel

Spezifikation: "Das Login-Formular soll ein Passwort-Feld haben."

Verifikation: ✅ Ist ein Passwort-Feld vorhanden? → JA = Verifikation erfolgreich

Verifikationsmethoden:

  • Statische Analyse: Code ohne Ausführung prüfen
  • Inspektion: Manuelle Überprüfung der Dokumente
  • Walkthrough: Schritt-für-Schritt Durchgang
  • Unit Tests: Einzelne Programmteile testen

✅ Validation - "Bauen wir das richtige Produkt?"

Validation überprüft, ob das System die Benutzerbedürfnisse erfüllt.

💡 Einfaches Beispiel

Benutzerbedürfnis: "Ich möchte mich schnell einloggen können."

Validation: ✅ Dauert der Login weniger als 3 Sekunden? → JA = Validation erfolgreich

Validationsmethoden:

  • Akzeptanztests: Kunde testet die Funktionen
  • Usability Tests: Benutzerfreundlichkeit prüfen
  • Beta-Tests: Echte Benutzer testen vorab
  • Feldtests: Test unter realen Bedingungen

🧪 Software-Testing

Systematisches Ausführen von Software um Fehler zu finden.

🎯 Definition of Done (DoD)

📋 Beispiel: Definition of Done für eine User Story

Eine Funktion gilt nur als "fertig", wenn:

  • ✅ Code geschrieben und getestet
  • ✅ Unit Tests geschrieben (mindestens 80% Abdeckung)
  • ✅ Code Review durchgeführt
  • ✅ Integration Tests erfolgreich
  • ✅ Dokumentation aktualisiert
  • ✅ Kunde hat die Funktion abgenommen

🔄 Continuous Integration/Continuous Deployment (CI/CD)

🔄 Continuous Integration

Code wird täglich zusammengeführt und automatisch getestet.

🚀 Continuous Deployment

Erfolgreiche Tests führen zu automatischer Bereitstellung.

🤖 Automatisierung

Alle Tests laufen automatisch bei jeder Code-Änderung.

⚡ Schnelles Feedback

Fehler werden sofort erkannt und gemeldet.

🔬 Testverfahren und -methoden

Systematisches Vorgehen beim Testen von Software

🎯 Testarten nach Zielrichtung

Testtyp Ziel Methode Beispiel
Funktionale Tests Funktionen prüfen Black-Box "Kann ich mich einloggen?"
Performance Tests Geschwindigkeit prüfen Last-Tests "Login unter 2 Sekunden?"
Security Tests Sicherheit prüfen Penetration "Können Hacker eindringen?"
Usability Tests Benutzerfreundlichkeit User-Tests "Finden Benutzer den Login?"

🔍 Black-Box vs. White-Box Testing

⬛ Black-Box Testing

Wie: Tester kennt den Code nicht

Fokus: Eingabe → Erwartete Ausgabe

Beispiel: Login mit falschen Daten → Fehlermeldung erwartet

⬜ White-Box Testing

Wie: Tester kennt den Code

Fokus: Interne Logik und Pfade

Beispiel: Werden alle if-else Zweige durchlaufen?

📊 Testabdeckung (Test Coverage)

📈 Was ist Testabdeckung?

Testabdeckung misst, welcher Anteil des Codes durch Tests geprüft wurde.

💻 Beispiel: Testabdeckung berechnen
Gesamte Codezeilen: 1000
Getestete Codezeilen: 800
Testabdeckung = 800/1000 = 80%

⚠️ Wichtiger Hinweis

100% Testabdeckung bedeutet NICHT 100% fehlerfreie Software! Es bedeutet nur, dass jede Codezeile mindestens einmal ausgeführt wurde.

🎯 Testabdeckungsarten:

  • Statement Coverage: Wurden alle Codezeilen ausgeführt?
  • Branch Coverage: Wurden alle Verzweigungen getestet?
  • Function Coverage: Wurden alle Funktionen aufgerufen?
  • Condition Coverage: Wurden alle Bedingungen getestet?

  • 🧪 Unit Tests für alle Berechnungen
  • 🔗 Integrationstests für Datenbank-Verbindungen

📈 QS-Metriken und KPIs

Um den Erfolg der Qualitätssicherung zu messen, brauchen wir konkrete Zahlen:

Metrik Beschreibung Zielwert Wie messen?
Defect Density Fehler pro Codezeile < 0.5 Fehler/1000 LOC Gefundene Fehler ÷ Lines of Code
Test Pass Rate Anteil erfolgreicher Tests > 95% Erfolgreiche Tests ÷ Alle Tests
Code Coverage Testabdeckung des Codes > 80% Automatische Messung durch Tools
Mean Time to Fix Durchschnittliche Reparaturzeit < 4 Stunden Zeit vom Fehlerreport bis Fix

🚨 Fehlermanagement

Wenn Fehler gefunden werden, müssen sie systematisch behandelt werden:

🔍 1. Fehler entdecken

Durch Tests, Code Reviews oder Benutzer-Feedback

📝 2. Fehler dokumentieren

In Bug-Tracking-System erfassen (z.B. Jira, GitHub Issues)

⚖️ 3. Fehler bewerten

Priorität festlegen: Kritisch, Hoch, Mittel, Niedrig

🔧 4. Fehler beheben

Entwickler behebt den Fehler und testet die Lösung

🐛 Beispiel: Fehlerklassifikation

Priorität Beispiel Reaktionszeit
🔴 Kritisch System stürzt ab, Datenverlust Sofort (< 2h)
🟡 Hoch Login funktioniert nicht Heute (< 8h)
🔵 Mittel Tippfehler in Texten Diese Woche
⚪ Niedrig Verbesserungsvorschlag Nächste Version

🔧 QS für Embedded Systems (Arduino, Mikrocontroller)

Bei Embedded Systems geht es oft um kritische Anwendungen - ein Fehler kann gefährlich sein!

🎯 Besondere Anforderungen:

  • ⚡ Real-Time Constraints: Reaktionen müssen in bestimmter Zeit erfolgen
  • 🔋 Energieeffizienz: Minimaler Stromverbrauch
  • 🌡️ Umgebungsbedingungen: Temperatur, Feuchtigkeit, Vibrationen
  • 🔒 Sicherheit: System darf nie in gefährlichen Zustand gelangen

🚗 Beispiel: QS für Arduino-basierte Ampelsteuerung

Arduino Code-Struktur für Tests:

🚦 Setup & Konfiguration
void setup() {
  // LED-Pins konfigurieren
  pinMode(RED_LED, OUTPUT);
  pinMode(YELLOW_LED, OUTPUT);
  pinMode(GREEN_LED, OUTPUT);
  
  // Sensor-Pins konfigurieren
  pinMode(SENSOR_PIN, INPUT);
  
  // Serial für Debugging
  Serial.begin(9600);
  
  // Initialer Zustand: Rot
  setTrafficLight(RED);
}
🔄 Hauptprogramm-Loop
void loop() {
  // Sensor-Eingaben lesen
  boolean carDetected = digitalRead(SENSOR_PIN);
  
  // Ampel-Logik ausführen
  switch(currentState) {
    case RED:
      if (carDetected && timeInState > MIN_RED_TIME) {
        changeToGreen();
      }
      break;
      
    case GREEN:
      if (!carDetected || timeInState > MAX_GREEN_TIME) {
        changeToYellow();
      }
      break;
      
    case YELLOW:
      if (timeInState > YELLOW_TIME) {
        changeToRed();
      }
      break;
  }
  
  updateTimer();
  checkSafety();
}
🛡️ Sicherheitsfunktionen
void checkSafety() {
  // Niemals beide Richtungen grün!
  if (digitalRead(GREEN_LED_1) && digitalRead(GREEN_LED_2)) {
    // NOTFALL: Alle Ampeln auf Rot
    emergencyStop();
    Serial.println("FEHLER: Beide Richtungen grün!");
  }
  
  // Watchdog-Timer prüfen
  if (millis() - lastHeartbeat > WATCHDOG_TIMEOUT) {
    emergencyStop();
    Serial.println("FEHLER: System hängt!");
  }
}

void emergencyStop() {
  // Alle Ampeln auf Rot setzen
  setAllLights(RED);
  // Warnsignal aktivieren
  digitalWrite(WARNING_LED, HIGH);
}

Testfälle für Arduino-Ampel:

  • ✅ Funktional: Alle LED-Kombinationen funktionieren
  • ⏱️ Timing: Phasenwechsel erfolgen in korrekter Zeit
  • 🚨 Notfall: Emergency-Stop bei kritischen Fehlern
  • 🔋 Power: Verhalten bei Stromausfall
  • 🌡️ Umgebung: Funktion bei -20°C bis +60°C
  • ⚡ Performance: Reaktionszeit < 100ms

🗄️ Qualitätssicherung für Datenbanken

🎯 Datenbank-spezifische QS:

  • 🔐 Datenintegrität: Sind die Daten konsistent und korrekt?
  • ⚡ Performance: Sind Queries schnell genug?
  • 🔒 Security: Schutz vor SQL-Injection und unbefugtem Zugriff
  • 💾 Backup & Recovery: Können Daten wiederhergestellt werden?

🏫 Beispiel: QS für Schüler-Notenverwaltung-DB

Datenintegrität Tests:

🔍 SQL-Testqueries
-- Test: Noten müssen zwischen 1 und 6 liegen
SELECT * FROM noten WHERE note < 1 OR note > 6;
-- Ergebnis muss leer sein!

-- Test: Jeder Schüler muss einer gültigen Klasse zugeordnet sein
SELECT s.* FROM schueler s 
LEFT JOIN klassen k ON s.klassen_id = k.id 
WHERE k.id IS NULL;
-- Ergebnis muss leer sein!

-- Test: Keine doppelten Noten pro Schüler/Fach
SELECT schueler_id, fach_id, COUNT(*) 
FROM noten 
GROUP BY schueler_id, fach_id 
HAVING COUNT(*) > 1;

Performance Tests:

  • 📊 Notenliste für eine Klasse laden: < 200ms
  • 🔍 Schüler-Suche über alle Klassen: < 500ms
  • 📈 Zeugnis-Generierung: < 2s pro Schüler

Security Tests:

  • 🔐 Lehrer können nur ihre eigenen Klassen sehen
  • 🛡️ Direkte SQL-Eingaben werden blockiert
  • 🔑 Passwort-Hash wird nie im Klartext gespeichert

📝 Typische Abitur-Aufgaben zu QS

📋 Muster-Aufgabe 1: Testplan erstellen

Aufgabe: "Erstellen Sie einen Testplan für eine Online-Banking-Anwendung. Berücksichtigen Sie funktionale und nicht-funktionale Anforderungen."

✅ Musterlösung - Struktur:

📋 Testplan-Struktur
1. Testziele definieren
   - Funktionalität: Überweisungen, Kontostand, Login
   - Sicherheit: Schutz vor unbefugtem Zugriff
   - Performance: Antwortzeiten unter 3 Sekunden
   - Usability: Intuitive Bedienung für alle Altersgruppen

2. Teststufen festlegen
   - Unit Tests: Einzelne Funktionen (z.B. Betragsberechnung)
   - Integrationstests: Datenbank-Anbindung
   - Systemtests: Gesamte Anwendung
   - Akzeptanztests: Test durch echte Bank-Kunden

3. Testmethoden auswählen
   - Black-Box: Funktionalität aus Benutzersicht
   - White-Box: Interne Programmlogik
   - Penetrationstests: Sicherheitslücken finden

4. Ressourcen planen
   - Testteam: 3 Tester für 2 Wochen
   - Testumgebung: Separater Server mit Testdaten
   - Tools: Selenium für Automatisierung

🎓 Abitur 2026 - Qualitätssicherung

Zusammenfassung: Qualitätssicherung ist ein systematischer Prozess zur Sicherstellung der Produktqualität durch Verifikation, Validation und Testing.
Praxis: Moderne QS nutzt agile Methoden, Automatisierung und kontinuierliche Integration.
Ziel: Fehlerfreie Software, die Benutzerbedürfnisse erfüllt und rechtzeitig geliefert wird.

💡 Merksatz für die Prüfung:

"Qualitätssicherung beginnt mit der Planung, läuft parallel zur Entwicklung und endet erst mit der Kundenzufriedenheit."
