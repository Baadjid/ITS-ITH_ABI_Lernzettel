📊 UML-Diagramme

Unified Modeling Language - Die Sprache der objektorientierten Modellierung

🎯 Was sind UML-Diagramme?

UML (Unified Modeling Language) ist eine standardisierte Sprache zur Visualisierung, Spezifikation, Konstruktion und Dokumentation von Softwaresystemen. UML-Diagramme helfen uns dabei, komplexe Software-Strukturen und -Abläufe grafisch darzustellen, bevor wir sie programmieren.

Warum UML verwenden?

  • 📈 Visualisierung: Komplexe Systeme werden verständlich dargestellt
  • 🗣️ Kommunikation: Einheitliche "Sprache" für Entwickler
  • 📋 Planung: Struktur wird vor der Implementierung durchdacht
  • 📚 Dokumentation: Code wird durch Diagramme erklärt

🔍 Definition: UML

UML (Unified Modeling Language) ist eine grafische Modellierungssprache, die verschiedene Diagrammtypen bereitstellt, um sowohl die statische Struktur als auch das dynamische Verhalten von objektorientierten Systemen zu beschreiben.

🎨 Die vier wichtigsten UML-Diagrammtypen im Abitur

Für das Abitur in Informationstechnik sind vier UML-Diagrammtypen besonders wichtig:

🏗️ Klassendiagramm

Zweck: Zeigt die statische Struktur des Systems

Inhalt: Klassen, Attribute, Methoden und ihre Beziehungen

Verwendung: Planung der Code-Struktur

📸 Objektdiagramm

Zweck: Zeigt einen "Schnappschuss" zur Laufzeit

Inhalt: Konkrete Objekte mit aktuellen Attributwerten

Verwendung: Verstehen der Objekt-Beziehungen

📞 Sequenzdiagramm

Zweck: Zeigt zeitliche Abfolge von Methodenaufrufen

Inhalt: Objekte und ihre Kommunikation über Zeit

Verwendung: Analyse von Programmabläufen

🔄 Zustandsdiagramm

Zweck: Zeigt mögliche Zustände eines Objekts

Inhalt: Zustände und Übergänge zwischen ihnen

Verwendung: Modellierung von Systemverhalten

🏗️ Klassendiagramm - Der Bauplan des Systems

🔍 Was ist ein Klassendiagramm?

Ein Klassendiagramm zeigt die statische Struktur eines Systems. Es stellt Klassen als "Baupläne" für Objekte dar und zeigt deren Attribute, Methoden sowie die Beziehungen zwischen den Klassen.

📋 Aufbau einer Klasse im UML-Klassendiagramm

Eine Klasse wird als Rechteck dargestellt, das in drei Bereiche unterteilt ist:

Klassenname
+ attribut1: String
- attribut2: int = 2
# attribut3: boolean
+ methode1(): String
+ setMethode(param1: String): void
- privateMethode(): int

🎯 Die drei Bereiche einer Klasse:

  1. Klassenname: Steht im Singular, beginnt mit Großbuchstaben
  2. Attribute: Eigenschaften der Klasse (Variablen)
  3. Methoden: Funktionen der Klasse (was sie kann)

👁️ Sichtbarkeitsmodifikatoren

Vor jedem Attribut und jeder Methode steht ein Symbol, das die Sichtbarkeit anzeigt:

Symbol Bedeutung Java-Keyword Zugriff von
+ public (öffentlich) public Überall
- private (privat) private Nur innerhalb der Klasse
# protected (geschützt) protected Klasse und Subklassen
~ package (paketbezogen) (kein Keyword) Innerhalb des Pakets

📝 Notation von Attributen und Methoden

⚠️ Wichtige Notation:

  • Attribute: sichtbarkeit name: Datentyp = Anfangswert
  • Methoden: sichtbarkeit name(parameter: Typ): Rückgabetyp
  • Unterstrichen: static (Klassenmethode/-attribut)
  • Kursiv: abstract (abstrakte Klasse/Methode)
  • {final}: final (nicht veränderbar)

📖 Beispiel einer vollständigen Klasse:

Auto
- marke: String
- geschwindigkeit: int = 0
+ maxGeschwindigkeit: int = 200
- anzahlAutos: int = 0
+ Auto(marke: String)
+ beschleunigen(): void
+ bremsen(): void
+ getGeschwindigkeit(): int
- motorStarten(): boolean
+ getAnzahlAutos(): int
public class Auto { // private Attribute private String marke; private int geschwindigkeit = 0; // public Attribut public int maxGeschwindigkeit = 200; // static Attribut (unterstrichen in UML) private static int anzahlAutos = 0; // Konstruktor public Auto(String marke) { this.marke = marke; anzahlAutos++; } // public Methoden public void beschleunigen() { /* ... */ } public int getGeschwindigkeit() { return geschwindigkeit; } // private Methode private boolean motorStarten() { /* ... */ } // static Methode (unterstrichen in UML) public static int getAnzahlAutos() { return anzahlAutos; } }

🔗 Beziehungen zwischen Klassen

Klassen stehen nicht allein - sie haben Beziehungen zueinander. Die wichtigsten Beziehungstypen sind:

🧬 Vererbung (Generalization)

Subklasse ——————————▷ Superklasse

Bedeutung: "ist-ein" Beziehung

Beispiel: Auto erbt von Fahrzeug

Java: class Auto extends Fahrzeug

📋 Interface-Implementierung

Klasse ——————————▷ Interface
(gestrichelte Linie)

Bedeutung: Klasse implementiert Interface

Beispiel: Auto implementiert Bewegbar

Java: class Auto implements Bewegbar

🤝 Assoziation

Klasse1 ——————————— Klasse2

Bedeutung: "kennt" oder "verwendet" Beziehung

Beispiel: Person kennt Auto

Java: Person hat Auto-Referenz

🔓 Aggregation

Klasse1 ◇——————————— Klasse2

Bedeutung: "hat-ein" Beziehung (schwach)

Beispiel: Abteilung hat Mitarbeiter

Besonderheit: Teile können ohne Ganzes existieren

🔒 Komposition

Klasse1 ◆——————————— Klasse2

Bedeutung: "hat-ein" Beziehung (stark)

Beispiel: Haus hat Zimmer

Besonderheit: Teile können nicht ohne Ganzes existieren

📦 Abhängigkeit

Klasse1 - - - - - - - - - →ᐟ Klasse2
(gestrichelte Linie)

Bedeutung: temporäre Verwendung

Beispiel: Methode verwendet Parameter-Objekt

Java: Objekt wird als Parameter übergeben

📸 Objektdiagramm - Momentaufnahme zur Laufzeit

🔍 Was ist ein Objektdiagramm?

Ein Objektdiagramm zeigt einen "Schnappschuss" des Systems zu einem bestimmten Zeitpunkt. Es stellt konkrete Objekte (Instanzen) mit ihren aktuellen Attributwerten dar, nicht die abstrakten Klassen.

📋 Aufbau eines Objekts im UML-Objektdiagramm

Ein Objekt wird ähnlich wie eine Klasse dargestellt, aber mit wichtigen Unterschieden:

objektName: Klassenname
marke: String = "BMW"
geschwindigkeit: int = 60
maxGeschwindigkeit: int = 200

🎯 Wichtige Merkmale des Objektdiagramms:

  • Objektname: konkrete Bezeichnung, z.B. "meinAuto"
  • Klassenname: wird nach dem Doppelpunkt angegeben
  • Unterstreichung: der gesamte Objektname ist unterstrichen
  • Attributwerte: konkrete Werte statt Datentypen
  • Keine Methoden: Objektdiagramme zeigen nur Attribute

📖 Praktisches Beispiel:

Gegeben sei die Klasse Auto aus dem vorherigen Beispiel. Zur Laufzeit könnten drei konkrete Auto-Objekte so aussehen:

bmwX5: Auto
marke = "BMW"
geschwindigkeit = 80
maxGeschwindigkeit = 250
vwGolf: Auto
marke = "Volkswagen"
geschwindigkeit = 0
maxGeschwindigkeit = 180
audiA4: Auto
marke = "Audi"
geschwindigkeit = 120
maxGeschwindigkeit = 220

⚠️ Objektdiagramm vs. Klassendiagramm:

Klassendiagramm Objektdiagramm
Zeigt abstrakte Struktur Zeigt konkreten Zustand
Datentypen (String, int) Konkrete Werte ("BMW", 80)
Methoden sichtbar Nur Attribute sichtbar
Zur Designzeit Zur Laufzeit

📞 Sequenzdiagramm - Kommunikation über die Zeit

🔍 Was ist ein Sequenzdiagramm?

Ein Sequenzdiagramm beschreibt die zeitliche Abfolge von Nachrichten (Methodenaufrufen) zwischen Objekten. Es zeigt, wie Objekte miteinander kommunizieren, um eine bestimmte Aufgabe zu erfüllen.

📋 Aufbau eines Sequenzdiagramms

objektA:KlasseA
objektB:KlasseB
objektC:KlasseC

🎯 Die wichtigsten Elemente:

  • Objekte: Rechtecke oben mit "objektName:Klassenname"
  • Lebenslinie: Senkrechte gestrichelte Linie unter jedem Objekt
  • Aktivitätsbalken: Schmale Rechtecke auf der Lebenslinie (Methode ist aktiv)
  • Nachrichten: Pfeile zwischen den Lebenslinien (Methodenaufrufe)
  • Rückgaben: Gestrichelte Pfeile zurück (return-Werte)

📨 Arten von Nachrichten

🔄 Synchroner Aufruf

————————————————————————▶

Bedeutung: Aufrufer wartet auf Antwort

Beispiel: methode(parameter)

Pfeil: Durchgezogene Linie mit gefüllter Pfeilspitze

⚡ Asynchroner Aufruf

————————————————————————▷

Bedeutung: Aufrufer wartet nicht

Beispiel: Thread-Start, Event-Handling

Pfeil: Durchgezogene Linie mit offener Pfeilspitze

📤 Rückgabe

◀———————————————————————— (gestrichelte Linie)

Bedeutung: Return-Wert oder Bestätigung

Beispiel: return result;

Pfeil: Gestrichelte Linie mit offener Pfeilspitze

🎯 Selbstaufruf

|————————————————————————⤷

Bedeutung: Objekt ruft eigene Methode auf

Beispiel: this.privateMethode()

Pfeil: Schleife zurück zum selben Objekt

📖 Praktisches Beispiel: Online-Bestellung

Stellen wir uns vor, ein Kunde bestellt online ein Produkt. Das Sequenzdiagramm könnte so aussehen:

kunde:Kunde
shop:OnlineShop
lager:Lager
zahlung:Zahlung
1. kunde ——————▶ shop: bestelleProdukt(produktId)
2. shop ——————▶ lager: prüfeVerfügbarkeit(produktId)
3. lager - - - - ▶ shop: return: true
4. shop ——————▶ zahlung: verarbeiteZahlung(betrag)
5. zahlung - - - ▶ shop: return: erfolgreich
6. shop ——————▶ lager: reserviereProdukt(produktId)
7. shop - - - - - ▶ kunde: return: bestellbestätigung

💡 Tipps zum Lesen von Sequenzdiagrammen:

  • Zeit: Von oben nach unten = zeitlicher Ablauf
  • Reihenfolge: Nachrichten sind nummeriert (1, 2, 3, ...)
  • Aktivität: Rechtecke auf Lebenslinien = Methode wird ausgeführt
  • Return: Gestrichelte Pfeile zeigen Rückgabewerte

🔄 Zustandsdiagramm - Verhalten über die Zeit

🔍 Was ist ein Zustandsdiagramm?

Ein Zustandsdiagramm beschreibt die möglichen Zustände eines Objekts und die Bedingungen, unter denen es von einem Zustand in einen anderen wechselt. Es zeigt das dynamische Verhalten eines Systems.

📋 Grundelemente eines Zustandsdiagramms

⚫ Startzustand

⚫————————▶ [Zustand]

Symbol: Schwarzer Kreis

Bedeutung: Anfangszustand des Systems

Regel: Genau einer pro Diagramm

🔚 Endzustand

[Zustand] ————————▶ ⦿

Symbol: Schwarzer Kreis mit Rand

Bedeutung: System ist beendet

Regel: Kann mehrere geben

📦 Zustand

┌─────────────┐
│ Zustand │
│ entry/aktion│
│ do/verhalten│
└─────────────┘

Symbol: Abgerundetes Rechteck

Inhalt: Name, entry/exit-Aktionen

➡️ Transition

[Zustand1] ————————▶ [Zustand2]
ereignis[bedingung]/aktion

Symbol: Pfeil zwischen Zuständen

Label: ereignis[bedingung]/aktion

📖 Praktisches Beispiel: Ampelsteuerung

Eine Verkehrsampel ist ein perfektes Beispiel für ein Zustandsdiagramm:

⚫ ——————————▶ [Rot]
Rot
entry/rotAn()
do/warten()
timer(30s) ————————▶
Grün
entry/grünAn()
do/fahrenErlaubt()
⬆
timer(5s)/gelbAn()
⬇
Gelb
entry/gelbAn()
do/bremsbereit()
timer(3s) ————————▶ zurück zu Rot

🔄 Implementierung in Code (Beispiel Arduino)

// Zustandsdiagramm in C++ implementiert enum AmpelZustand { ROT, GELB, GRUEN }; AmpelZustand aktuellerZustand = ROT; unsigned long letzteZeit = 0; void ampelSteuern() { unsigned long aktuelleZeit = millis(); switch(aktuellerZustand) { case ROT: // entry/rotAn() digitalWrite(ROT_PIN, HIGH); digitalWrite(GELB_PIN, LOW); digitalWrite(GRUEN_PIN, LOW); // Transition nach 30 Sekunden if(aktuelleZeit - letzteZeit > 30000) { aktuellerZustand = GRUEN; letzteZeit = aktuelleZeit; } break; case GRUEN: // entry/grünAn() digitalWrite(GRUEN_PIN, HIGH); digitalWrite(ROT_PIN, LOW); // Transition nach 25 Sekunden if(aktuelleZeit - letzteZeit > 25000) { aktuellerZustand = GELB; letzteZeit = aktuelleZeit; } break; case GELB: // entry/gelbAn() digitalWrite(GELB_PIN, HIGH); digitalWrite(GRUEN_PIN, LOW); // Transition nach 5 Sekunden if(aktuelleZeit - letzteZeit > 5000) { aktuellerZustand = ROT; letzteZeit = aktuelleZeit; } break; } }

📁 Pakete - Organisation von Klassen

🔍 Was sind Pakete?

Pakete fassen zusammengehörige Klassen zu logischen Einheiten zusammen. Sie helfen dabei, große Projekte zu strukturieren und Namenskonflikte zu vermeiden.

📋 Darstellung von Paketen

Pakete werden in UML wie ein Aktenordner dargestellt:

fahrzeuge
Auto
Fahrrad
Motorrad

🎯 Eigenschaften von Paketen:

  • Name: Wird groß geschrieben (wie Klassennamen)
  • Inhalt: Können Klassen und andere Pakete enthalten
  • Hierarchie: Pakete können verschachtelt werden
  • Namespace: Jede Klasse gehört zu genau einem Paket

📖 Beispiel einer Paketstruktur:

// Java-Paketstruktur package de.schule.fahrzeuge; public class Auto { // Klassen-Inhalt... } // In einer anderen Datei: package de.schule.personen; import de.schule.fahrzeuge.Auto; // Import aus anderem Paket public class Person { private Auto meinAuto; // Verwendung der importierten Klasse }

📚 Zusammenfassung: UML-Diagramme im Überblick

🎯 Die vier wichtigsten UML-Diagramme für das Abitur:

Diagrammtyp Zweck Zeigt Wann verwenden
Klassendiagramm Statische Struktur Klassen, Attribute, Methoden, Beziehungen Planung der Software-Architektur
Objektdiagramm Momentaufnahme Konkrete Objekte mit Werten Verstehen der Laufzeit-Struktur
Sequenzdiagramm Zeitlicher Ablauf Methodenaufrufe über Zeit Analyse von Programmabläufen
Zustandsdiagramm Verhalten modellieren Zustände und Übergänge Steuerungslogik, Automaten

💡 Prüfungstipps für UML-Diagramme:

  • Sichtbarkeit: + (public), - (private), # (protected), ~ (package)
  • Notation: attribut:Typ, methode(parameter:Typ):Rückgabetyp
  • Beziehungen: Pfeile richtig zeichnen (gefüllt/offen, durchgezogen/gestrichelt)
  • Objekte: Immer unterstrichen, konkrete Werte statt Typen
  • Sequenz: Zeit läuft von oben nach unten
  • Zustand: Startzustand (⚫), Endzustand (⦿), Transitionen beschriften
